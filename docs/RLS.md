# Row-Level Security Reference

How the CID Portal's database-driven security wall is built: the `private.*` helper functions, the column-freeze triggers, the `SECURITY DEFINER` conventions, and the live test suites that prove both denial and approval paths. Policy sources of truth: [`supabase/schema-snapshot.sql`](../supabase/schema-snapshot.sql) and [`supabase/migrations/`](../supabase/migrations/). Who-may-do-what lives in [AUTHORIZATION.md](AUTHORIZATION.md); the reviewer checklist in [SECURITY-REVIEW.md](SECURITY-REVIEW.md).

## 1. Core authorization helpers (`private` schema)

All are `SECURITY DEFINER` with `set search_path to ''` and key on `auth.uid()` — the signed-in human, never a client-supplied identity.

## 2. Profile visibility and the freeze triggers

- **`profiles_sel`**: your own row, or any row if you are an active member. But the **`email` column is grant-revoked**: [`20260708140000_restrict_profile_email.sql`](../supabase/migrations/20260708140000_restrict_profile_email.sql) removed the table-level SELECT grant and re-granted every column *except* `email`; Command reads addresses via the `admin_member_emails()` RPC. Column grants do **not** extend to columns added later — `removed_at` and the `login_denied*` columns were granted explicitly in their own migrations.
- **Privileged-column freezes** — three triggers guard `profiles`:
  - `private.guard_profile()` (**definer**): `is_owner` immutable from the client for everyone; a non-command member cannot change their own `role`/`active`/`division`.
  - `private.block_direct_login_denied()` (**non-definer**): freezes `login_denied*` against all direct client writes ([`20260713090000_login_denial.sql`](../supabase/migrations/20260713090000_login_denial.sql)).
  - `private.block_direct_privileged_profile()` (**non-definer**): freezes `role`/`division`/`active`/`is_owner`/`removed_at`/`is_test`/`is_system` for *all* direct client writes — closing the old `profiles_command` bypass where any command member could UPDATE those columns via PostgREST ([`20260718010000_unified_role_policy.sql`](../supabase/migrations/20260718010000_unified_role_policy.sql); `is_test` joined in [`20260719020000_hide_test_fixtures.sql`](../supabase/migrations/20260719020000_hide_test_fixtures.sql), `is_system` in [`20260726010000_phase_b_permanent_deletion.sql`](../supabase/migrations/20260726010000_phase_b_permanent_deletion.sql)).
- **System rows** (`profiles.is_system` — currently only the Phase B 'Deleted Member' tombstone): `profiles_sel` hides them from every viewer except the owner, the same chokepoint that hides `is_test` fixtures from real members. The client mirrors the exclusion in `activeProfiles()` so owner sessions never see the tombstone in pickers.
- **Why the freeze triggers are NON-definer**: inside a `SECURITY DEFINER` trigger, `current_user` is the function owner, so the trigger could not tell a raw PostgREST UPDATE from a legitimate definer RPC. As plain (invoker) triggers, `current_user` is `authenticated`/`anon` for direct PostgREST writes (frozen) and the function owner when a definer RPC performs the update (passes through). The same pattern protects `reports` (`block_direct_report_finalize`), `cases` (`block_direct_signoff`, and `block_direct_case_bureau` freezing `bureau`/`originating_bureau` — the authorized path is the `case_reassign_bureau` RPC, DD+/Owner, [`20260725010000_case_bureau_reassignment.sql`](../supabase/migrations/20260725010000_case_bureau_reassignment.sql)), and the membership/justice request guard triggers.

## 3. Case, evidence, justice, and sealed-request access

- **Cases and satellites**: every case-scoped table (`reports`, `evidence`, `case_tasks`, `case_messages`, `case_intel_links`, `mo_profiles`, `rico_cases`+`predicate_acts`, `raid_compensations`, …) delegates to `private.can_access_case(case_id)`. `custody_chain` derives access from its parent evidence row and is append-only (no update/delete policies).
- **Justice tables**: SELECT-only for clients; **no INSERT/UPDATE/DELETE grants exist** on any legal table — the transactional definer RPCs are the only write path ([DOJ-INTEGRATION.md](DOJ-INTEGRATION.md)).
- **Sealed legal requests are undiscoverable by construction**: `lr_sel` → `private.can_view_legal_request()` limits visibility to creator, active participants, Owner, and DA/AG oversight of DOJ-submitted requests (CID case members see `standard` classification only). Search never widens that: `legal_search()` and the legal union in `search_all()` are **SECURITY INVOKER**, so every result passes through the caller's own SELECT policy — no hits, no counts, no suggestions for a sealed request ([`20260714050000_legal_search_cleanup.sql`](../supabase/migrations/20260714050000_legal_search_cleanup.sql), [`20260715020000_search_all_legal.sql`](../supabase/migrations/20260715020000_search_all_legal.sql)). Notifications in the legal workflow are sealed-safe (recipient-scoped, header text only), and classified narratives are never indexed.

## 4. Owner-only functions

`private.is_owner()` gates `audit_log` reads (`audit_sel`), `feedback_meta` triage, `client_errors` reads/deletes, `owner_security_overview()`, and the Phase B permanent-deletion surface: `permanent_delete_preview/_arm/_execute()` (arm/execute additionally require a **fresh session** via `private.assert_fresh_session()` — the caller's `auth.sessions` row must be under 5 minutes old) and `deleted_member_ledger` reads (`dml_sel`; the ledger has **zero** client write policies — only `permanent_delete_execute` writes it). `private.is_owner_maintenance()` (flag-only) gates exactly two RPCs: `import_legal_warrant()` and `import_rollback_by_key()`. `app_secrets` has RLS enabled with zero policies — invisible to every client role by design; `deletion_tokens` follows the same pattern (zero policies, client grants revoked — the arm/execute RPCs are the only path). See [AUTHORIZATION.md](AUTHORIZATION.md) §4 for the full arm/execute protocol and blocker list ([`20260726010000_phase_b_permanent_deletion.sql`](../supabase/migrations/20260726010000_phase_b_permanent_deletion.sql)).

## 5. Test fixtures

- Live-suite accounts follow the pattern `rls-test-*@cidportal.test` (roster in [`tests/rls/README.md`](../tests/rls/README.md)). They hold no command role; the owner fixture carries only `is_owner`.
- **Notification fan-out exclusion**: RPCs that fan out to real command staff skip the fan-out when the acting account is a fixture — `membership_request_submit()` ([`20260713080000_test_applicant_notification_guard.sql`](../supabase/migrations/20260713080000_test_applicant_notification_guard.sql)) and `private.transfer_notify()` ([`20260718020000_officer_transfers.sql`](../supabase/migrations/20260718020000_officer_transfers.sql)) — so test runs never ping real officers.
- **Gated fixture RPCs**: `rls_test_cleanup()` may only be called *by* a fixture and deletes only fixture-authored rows; `rls_test_reset_member()` requires **both** caller and target to be fixture accounts, so production profiles are out of reach by construction ([`20260718020000_officer_transfers.sql`](../supabase/migrations/20260718020000_officer_transfers.sql)). `rls_test_set_signoff(p_case, p_status, p_stage)` ([`20260721040001_rls_test_signoff_helper.sql`](../supabase/migrations/20260721040001_rls_test_signoff_helper.sql)) places a case at a sign-off state deterministically for `v119`; it requires the **caller** to be a fixture **and** the case to be fixture-owned, so it can never touch a real case, and it exists only because `private.signoff_pick` selects deputy/director *globally* (which could otherwise route a test into a real reviewer). `rls_test_spawn_disposable(p_suffix)` ([`20260726010000_phase_b_permanent_deletion.sql`](../supabase/migrations/20260726010000_phase_b_permanent_deletion.sql)) creates a synthetic, banned, never-logs-in `rls-test-disposable-*` member so `v125` can arm+execute a **real** permanent deletion without touching the standing fixtures; same caller gate as `rls_test_cleanup()`, which also sweeps leftover disposables, fixture-armed `deletion_tokens`, and disposable `deleted_member_ledger` rows.

## 6. SECURITY DEFINER rules

Every workflow RPC follows the same contract:

- **Why definer is required**: the RPC must atomically mutate rows the caller's own RLS would refuse (e.g. flipping `profiles.active` on approval, writing `role_events`, advancing a legal request) while validating the caller's authority *inside* the function. Non-definer is reserved for the freeze triggers (§2) and invoker-scoped search (§3).
- **Fixed search path**: every function declares `set search_path to ''` and schema-qualifies all references — no resolution hijacking.
- **Revoke-then-grant**: `revoke all on function ... from public;` then `grant execute on function ... to authenticated, service_role;` — anonymous execution is never possible.
- **Row locking**: decision RPCs take `select ... for update` on the row being decided (and, where relevant, the target profile) before validating state — `review_membership_request`, `change_member_role`, the `*_transfer` RPCs, `report_finalize` (since [`20260715040000_v114_hardening.sql`](../supabase/migrations/20260715040000_v114_hardening.sql)), the sign-off decision RPCs `signoff_submit` / `signoff_decide` / `signoff_owner_action` / `signoff_command_override` (since [`20260721040000_signoff_integrity.sql`](../supabase/migrations/20260721040000_signoff_integrity.sql)), and `warrant_set_status` (since [`20260722010000_warrant_lifecycle_integrity.sql`](../supabase/migrations/20260722010000_warrant_lifecycle_integrity.sql)) — so two concurrent human decisions cannot both apply. The loser re-reads the post-first-commit row and its precondition fails, raising an **application conflict (SQLSTATE `P0001`, "…reload and retry")** — a clear conflict, not a silent no-op, and *not* a lock-not-available error (`55P03`), which only applies to `NOWAIT`.
- **Named-actor validation**: the first statements load the caller's profile by `auth.uid()` and raise on missing/inactive/unauthorized — the caller can never assert an identity or a role in a parameter (hidden-field role smuggling is tested and rejected).
- **Null-safe authority predicates**: owner/role checks compare with `is not distinct from` (or `coalesce(<pred>, false)`), never a bare `=` against a nullable column. A bare `v_uid = c.lead_detective_id` yields `NULL` (not `FALSE`) when `lead_detective_id` is `NULL`, and `if not (… and NULL)` would fall through and admit a non-owner. Fixed for the sign-off owner/override predicates in [`20260721040000_signoff_integrity.sql`](../supabase/migrations/20260721040000_signoff_integrity.sql); a `bureau_lead` at the deputy stop-point of an unowned-lead case is the regression pinned by `v119`.

### Sign-off integrity (case_signoff_history)

- **History is RPC-only** since [`20260721040000_signoff_integrity.sql`](../supabase/migrations/20260721040000_signoff_integrity.sql): the client `INSERT` policy (`csh_ins`) is dropped and `INSERT/UPDATE/DELETE/TRUNCATE` are revoked from `authenticated` (and all grants from `anon`). Only the SECURITY DEFINER sign-off RPCs write it. Reads stay open to case participants via `csh_sel`.
- **Structured provenance**: every RPC-written row carries `actor_id` (a real uuid, not just a name snapshot), `from_status`, and a `source` of `submit | reviewer | owner | command_override`. Owner vs. command-override is therefore a **structural** distinction, not a free-text convention. Historical rows keep `actor_id` NULL — no name-based backfill.
- **Strict owner + narrow override**: `signoff_owner_action` admits only the case owner (lead detective **or** original submitter). A separate `signoff_command_override(p_case, p_action, p_reason)` lets **Deputy Director / Director / Owner (never a Bureau Lead, never rank-and-file)** act in the owner's place when the owner is unavailable; it requires a non-blank reason and is audited as `source='command_override'` with the reason in `note`.

### SQL function header documentation convention

The standing convention for any **new or materially modified** SQL function: open the definition with a comment block documenting —

```sql
-- Purpose:        what the function does, in one or two lines
-- Caller:         who invokes it (client surface, trigger, other RPC)
-- Authorization:  the exact check(s) — helper names, roles, matrix rules
-- Side effects:   rows written outside the primary table (notifications, role_events, …)
-- Audit behavior: what lands in audit_log / history tables, and when
-- Security notes: definer/invoker rationale, locking, freeze-trigger interactions
```

Existing functions carry this information as prose comments in their migrations; new work should use the structured block so a human reviewer can audit authorization at a glance.

## 7. How the live test suites verify the wall

The suites in [`tests/rls/`](../tests/rls/) (`rls.test.ts`, `legal.test.ts`, `v114`–`v135.test.ts`) sign in to the **live project** as the low-privilege fixtures using the anon key + password grant — **never a service key** — and assert both directions:

- **Denial paths**: bureau isolation (read/write/insert/child rows), deny-by-default for inactive accounts, the lockdown triggers, RPC caller checks (e.g. `signoff_decide` as a non-assignee), the `profiles.email` column grant, `is_owner` self-grant immunity, sealed-request undiscoverability (table, search, notifications), anonymous access. `v119` adds the sign-off integrity contract: concurrent-decision conflict (`P0001`, exactly one winner), direct `case_signoff_history` INSERT rejected, strict-owner negatives (non-owner detective **and** bureau_lead), and command-override authority (rank-and-file/bureau_lead/blank-reason all rejected; Director/Owner positive with `source='command_override'`).
- **Approval paths**: owner-positive blocks (triage writes, audit reads), the membership approval success path (atomic profile flip + notification), bureau-lead scoping that *succeeds* in the proper bureau, the DOJ review/approval routes, transfers. `v125` covers Phase B permanent deletion end-to-end against a disposable synthetic member: non-owner denial of all three RPCs, blank-reason/self/blocked-target refusals, wrong-confirmation and fabricated-token refusals, the full arm→execute success (profile + auth identity gone, owner-only ledger row with the reference map, ARMED/EXECUTED audit rows, single-use token), and the server-side tombstone hiding. The 5-minute stale-session denial cannot be simulated live (fixtures sign in fresh at suite start); it shares the fail-closed `private.assert_fresh_session()` predicate the token-refusal tests exercise. `v127` pins the case-access decision path: a requester cannot approve their own request, a bureau lead can (status flip + `case_access_grants` insert, after which the requester gains case visibility), a denied request grants nothing, and the `access_granted`/`access_denied` decision notifications are emittable only by a `private.can_grant_case` holder for that case.
- **Person intelligence** (`v128`, [`20260729010000_person_intelligence.sql`](../supabase/migrations/20260729010000_person_intelligence.sql)): the persons CHECK vocabularies and link-table constraints (self-link, inverse-duplicate, UNIQUE dupes), creator-or-command deletes on the person link tables, `search_persons`'s invoker-scoped fail-closed case-number matching across the bureau wall plus its anonymous denial, and `person_merge`'s command gate, blank-reason refusal, child repointing, tombstone semantics, and `PERSON_MERGED` audit row.
- **Approval-queue integrity** (`v129`, [`20260730010000_membership_reconciliation.sql`](../supabase/migrations/20260730010000_membership_reconciliation.sql)): direct activation via `assign_member` auto-reconciles a pending request (`approved`, `decided_by` = actor, appended internal `Auto-reconciled` note read through `admin_membership_requests`); `rejected`/`withdrawn` requests refuse direct activation (profile stays inactive) while already-active no-ops, role-only changes, and already-decided requests pass untouched; `review_membership_request` can still approve a legacy ghost (already-active applicant + pending request); and the `is_system` guards on `assign_member`/`admin_restore_member` refuse the tombstone. Non-command denial of `assign_member` is re-pinned.
- **Justice-request visibility & judiciary authority** (`v130`, [`20260731010000_justice_request_visibility.sql`](../supabase/migrations/20260731010000_justice_request_visibility.sql)): CID command can SELECT `justice_membership_requests` rows (`jmr_sel` + `is_command`) while a plain detective still sees none and `internal_decision_note` stays column-revoked for every client; command review and `admin_justice_membership_requests` are refused (visibility ≠ authority — judiciary approvals are Owner + AG); the AG can now approve a `judge` request (activating the judiciary membership without touching the CID profile) but still cannot approve an `attorney_general` request; and the new dual-active guard refuses approving an applicant who is an active CID member while corrections pass untouched. Writing this suite caught a LIVE regression: [`20260719020000`](../supabase/migrations/20260719020000_hide_test_fixtures.sql)'s redefinition of `admin_justice_membership_requests()` had dropped the [`20260714070000`](../supabase/migrations/20260714070000_legal_null_guards.sql) NULL-coalesce, letting any authenticated user read every justice request (incl. the revoked `internal_decision_note`) through the definer RPC — fixed in [`20260731020000_admin_justice_guard_fix.sql`](../supabase/migrations/20260731020000_admin_justice_guard_fix.sql), and the v130 pin now holds it down.
- **Document governance** (`v131`, [`20260801010000_document_governance.sql`](../supabase/migrations/20260801010000_document_governance.sql)): the full classification matrix in both directions (internal/restricted/command/justice — CID command denied on justice docs — /owner), draft invisibility to plain members, version-row visibility inheritance + the tightened version INSERT (pre-v131 any active member could fabricate history), workflow authority + reason requirements + the guard trigger keeping direct status writes inert, RPC-only version-specific acknowledgements with cross-user denial and manager-only aggregates, campaign publish authority + notification fan-out, strictly private bookmarks/reading state, RLS-scoped `search_documents` (no titles/snippets/counts of classified docs; anon revoked outright), reasoned safe restore, sync-resolution authority + NULL-safe conflict check, and the feedback `document` kind. Writing this suite caught three live defects pre-ship: missing version rows for direct-insert docs (fixed with `trg_document_initial_version`), a NULL three-valued `sync_status` check, and Supabase **default-privilege EXECUTE grants to anon** on new functions ('revoke from public' does not strip them — every document RPC now revokes anon explicitly).
- **Bureau-scoped document authority & suggestions** (`v132`, [`20260802010000_document_bureau_scope_suggestions.sql`](../supabase/migrations/20260802010000_document_bureau_scope_suggestions.sql)): an LSB Bureau Lead may edit an LSB-bureau doc but is denied a BCB-bureau doc, a NULL/division-wide doc, and any `command`-class doc (director/Owner pass those); the doc owner and division leadership pass as specified. Detective suggestions submit only against viewable docs (no existence leak), stay visible to their submitter and the doc's bureau-scoped managers but not to unrelated members; managers decide (note required for decline/needs-info), accepting leaves the SOP unedited, duplicates keep the row, and implementation links a version. Writing this suite caught a **live prod-breaking defect pre-commit**: the two new `private.` authority helpers used *inside RLS policy expressions* were `revoke ... from public` without a matching `grant execute ... to authenticated`, so every authenticated document write failed `permission denied for function` (RLS predicates run with the caller's EXECUTE right even for SECURITY DEFINER functions) — hotfixed in [`20260802020000_fix_document_authority_grants.sql`](../supabase/migrations/20260802020000_fix_document_authority_grants.sql).
- **Narcotics intelligence workspace** (`v133`, [`20260803010000_narcotics_intelligence.sql`](../supabase/migrations/20260803010000_narcotics_intelligence.sql)): the `private.guard_narcotic()` freeze (a detective's INSERT is forced to a provisional record — status `unidentified`, `restricted` cleared, `created_by` pinned to the caller and spoofs ignored; on UPDATE `created_by`/`merged_into` are immutable and non-managers cannot touch status/restricted/category/classification/charge_codes/reviewed_*); restricted rows blind to plain detectives; a detective edits only their own still-provisional row; Owner-only delete; child links/aliases/seizures inherit parent visibility (a link under a restricted substance disappears with it); RPC-only suggestions (submit works, direct INSERT denied, restricted-target returns "not found" with no leak, decide is manager-only with note-required decline); `merge_narcotics` repoints aliases/links/seizures to the survivor and tombstones the merged row (`status='merged'`, reason required); `resolve_provisional_narcotic` is manager-only and stamps the review columns; `search_narcotics` excludes tombstones and denies anon EXECUTE. The `private.guard_narcotic()` trigger is deliberately **NON-definer** (invoker) — a definer trigger's `current_user` would be the function owner, so the `authenticated`/`anon` freeze would never engage (same reason as the profiles/login freezes, §2). The two policy-referenced helpers (`can_manage_narcotics`/`can_edit_narcotics_intel`) carry the explicit `grant execute ... to authenticated` from the start (the v132 lesson).
- **Restricted street-value sales** (`v134`, [`20260804010000_narcotic_sales.sql`](../supabase/migrations/20260804010000_narcotic_sales.sql)): the three sales tables (`narcotic_sale_series`/`_observations`/`_stacks`) are restricted intelligence — SELECT/INSERT require `can_edit_narcotics_intel()` (senior_detective+ / Owner), so plain detectives (LSB/BCB fixtures) see **zero** rows even when filtering by a known seeded id; anon is denied; stacks inherit visibility through an EXISTS on a visible parent observation. Observation UPDATE needs `can_manage_narcotics()` or (`can_edit_narcotics_intel()` AND `state='draft'`) — a non-manager cannot mutate a confirmed observation; DELETE is Owner-only (series/observations) / manager-only (stacks). The two guard triggers (above) force `restricted=true` and block self-confirm.
